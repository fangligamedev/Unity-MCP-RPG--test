# NineKings V2 开发进度管理方法

更新时间：2026-03-19

## Overview

`NineKings V2` 的进度管理不是按“改了多少文件”或“写了多少行代码”来估算，而是按四个维度一起判断：

- 用户可见流程是否已经跑通
- 状态闭环是否已经完整
- 回归测试是否已经覆盖并通过
- 当前实现是否还存在明显风险或返工点

这意味着一个模块即使代码很多，只要用户还看不到、流程还没闭环、测试还没锁住，就不能算完成。

## Source of Truth

当前进度同步依赖四套真相源交叉校准：

- 运行时真实状态：`Run / Board / Hand / Battle`
- 表现层真实状态：`NineKingsV2ScenePresenter`
- 回归真实状态：`EditMode / PlayMode` 测试
- 人工总控视图：`NineKingsV2_StatusReport.md`

这四者分别回答不同问题：

- 状态机回答“系统现在处于哪个阶段”
- 表现层回答“玩家现在实际看到什么”
- 测试回答“这个结论是否稳定、是否会回归”
- 状态报告回答“整体项目推进到了哪一层”

## How Progress Is Estimated

进度估算采用混合模型，不是单一百分比。

- 表现层完成度：布局、镜头、拖拽、战斗构图、暗场、反馈是否接近目标观感
- 战斗表现完成度：列队、接敌、投射物、命中、死亡、掉落、结算节奏是否完整
- 玩法闭环完成度：开局、放牌、弃井、自动开战、奖励、下一年继续是否能持续跑
- 长局系统完成度：事件、敌王轮换、33 年、Final Battle、GM 是否接入
- 测试覆盖度：核心逻辑是否已有 EditMode / PlayMode 回归

一个模块只有同时满足下面三条，才会被计为“完成”：

- 用户可以直接验证
- 自动测试已经通过
- 没有明显未收口或高概率回归点

否则最多只能算：

- 已打底
- 已能跑
- 已部分完成

不能算真正完成。

## Sync Workflow

每轮实现后，进度同步固定按四段结构更新：

- 完成项
- 未完成项
- 风险
- 下一步

同步步骤固定如下：

1. 先看用户当前最直接可见的问题有没有解决
2. 再核对代码真实状态有没有形成稳定闭环
3. 然后跑 EditMode / PlayMode 测试修正主观判断
4. 最后把结论落到 `NineKingsV2_StatusReport.md`

这里会刻意区分两种状态：

- 已经能跑：功能存在，但还未收口
- 已经收口：功能、表现、测试都稳定

避免把“暂时可用”误判成“已经完成”。

## Why The Status Report Looks Detailed

`NineKingsV2_StatusReport.md` 之所以能写得比较细，不是因为记忆很多细节，而是因为它是三套证据交叉校准后的人工总控快照：

- 用户截图与当前最痛的可见问题
- 代码里的运行时状态结构与 phase 链
- EditMode / PlayMode 的实际回归结果

每次修改都会被映射回固定阶段框架：

- 表现层止血
- 战斗编队与可视化
- 战后节奏与暗场
- 王国接入与长局验证

所以状态报告不是流水账，而是一个“阶段框架 + 当前证据”的汇总结果。

## Update Rules

后续维护这套进度文档和状态报告时，固定遵守以下规则：

- 只记录稳定结论，不记录临时猜测
- 剩余工作始终按优先级排序
- 下一步只锁定一个主要 Sprint，不同时开太多战线
- 如果用户临时插入高优先级可见问题，先处理可见问题，再回到主计划
- 如果测试结果和主观判断冲突，以测试和运行时状态为准
- 每轮改动后，都要明确“这轮是在补底座，还是在收口主体功能”

这套方法的核心技术原理可以概括成一句话：

`开发进度 = 状态机闭环 + 表现层可验证结果 + 回归测试稳定性`

只要三角校验里有一角没有闭合，就不能把对应模块标为完成。
